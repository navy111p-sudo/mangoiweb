# 계획 — 화상수업 "끊겨도 안 아프게" (무중단 재연결)

> 작성 2026-07-24 · 상태: **착수 전(설계 확정용)** · 선행작업: `594e7861f` 까지 배포 완료
> 이 문서는 *구현 전에 합의할 것* 을 적은 것입니다. 코드는 아직 한 줄도 바꾸지 않았습니다.

## 1. 무엇을 고치려는가

지금은 **WebSocket 이 한 번만 깜빡여도 수업 전체가 리셋**됩니다.

필리핀 강사 회선에서는 이 깜빡임이 몇 분마다 일어나므로, 사용자에게는
"수업 중 갑자기 튕겼다 붙었다"로 보입니다. 이번(07-24) 작업으로 *끊긴 것을 알아채는 것*
(pong 워치독)과 *끊긴 티가 덜 나는 것*(재연결 중 타일 유지)까지는 했지만,
**끊김 자체의 비용**은 그대로입니다.

### 실제로 일어나는 일 (근거)

| # | 위치 | 동작 |
|---|---|---|
| 1 | `public/index.html:8874-8879` | 재연결이면 **`vcCleanupAllPeers()`** → 모든 PeerConnection `close()` + 모든 원격 타일 DOM 삭제 |
| 2 | `src/video-call-room.ts:62` | 서버가 **소켓마다 새 랜덤 userId** 발급 → 재연결 = 서버 입장에서 '다른 사람' |
| 3 | `src/video-call-room.ts:201` | 옛 소켓을 `close(4001)` → `handleLeaveRoom` → 상대들에게 **`user-left(dropped)`** 방송 |
| 4 | `public/index.html:11776-11785` | 재입장 후 `existing-users` 를 받아 **전원에게 `idx*300ms` 간격으로 offer 재발사** |

늘어나므로, 블립이 잦으면 수업 시간의 상당 부분이 암전입니다.

## 2. 핵심 통찰 — 미디어는 안 끊겼는데 우리가 죽이고 있다

**WebSocket 은 시그널링 전용입니다. 영상·음성은 WebRTC P2P 로 따로 흐릅니다.**

WS 가 끊겨도 PeerConnection 은 살아 있는 경우가 대부분입니다(ICE 는 UDP 홀펀칭이라
TCP 시그널링 경로와 독립). 즉 **1번(`vcCleanupAllPeers`)은 필요해서가 아니라,
2번(새 userId) 때문에 어쩔 수 없이 하는 청소**입니다.

> 정체성만 유지되면 청소할 이유가 사라집니다.

그래서 이 계획의 뼈대는 하나입니다:

**재연결 시 userId 를 그대로 유지하고, 살아 있는 PeerConnection 은 건드리지 않는다.**

---

## 3. 설계

### 3-A. 서버 — 같은 브라우저 탭이면 userId 를 물려준다

`clientId` 는 이미 있습니다. `vcClientId()`(`index.html:12998`)가 **탭 단위**로
`sessionStorage` 에 저장하므로 **재연결·새로고침에 그대로 유지**되고, 다른 탭/기기와는
절대 겹치지 않습니다. (이건 과거 "무한 킥 루프" 사고 때 이미 검증된 설계입니다.)

`handleJoinRoom`(`video-call-room.ts:182`)은 이미 같은 `clientId` 의 옛 소켓을 찾아
닫고 있습니다. **그 자리에서 옛 userId 를 물려받으면 됩니다.**

```
현재:  옛 소켓 발견 → close(4001) → user-left(dropped) 방송 → 새 userId 로 user-joined
계획:  옛 소켓 발견 → 옛 userId 를 새 소켓 attachment 에 복사
                    → 옛 소켓은 joined:false 로 표시한 뒤 close  ← user-left 를 쏘지 않음
                    → user-joined 도 쏘지 않음(이미 있는 사람이므로)
```

**상대방 입장에서는 아무 일도 일어나지 않습니다.** 타일도, PeerConnection 도 그대로입니다.

⚠️ 반드시 지킬 순서 (하나라도 어긋나면 유령/무음이 남습니다)
1. 옛 소켓 attachment 를 `{ joined:false, userId:'stale-'+userId }` 로 먼저 갱신
   — `webSocketClose` 가 `handleLeaveRoom` 을 건너뛰게 하고, `wsOf(userId)` 가
     새 소켓을 가리키게 하기 위함. **둘 다 필요합니다.**
2. 그 다음에 `close(4001)`
3. 새 소켓 attachment 에 물려받은 `userId` 기록
4. `room-joined` 는 물려받은 userId 로 회신 (클라이언트가 이걸로 판단함 — 3-B 참조)
5. `user-joined` 브로드캐스트 **생략**, 대신 `existing-users` 는 정상 전송

### 3-B. 클라이언트 — 정체성이 유지됐으면 청소하지 않는다

클라이언트는 `room-joined` 의 `userId`(`index.html:11769`)로 판단할 수 있습니다.

```
재연결 onopen        → vcCleanupAllPeers() 를 '즉시' 부르지 않고 보류
room-joined 수신     → 새 userId === 직전 vcUserId ?
                        예 → 청소 생략. 살아있는 PC 유지.
                        아니오 → 지금처럼 vcCleanupAllPeers() (기존 동작 = 안전한 쪽)
```

⚠️ `existing-users` 처리도 함께 고쳐야 합니다(`index.html:11776`).
지금은 **무조건 전원에게 offer 를 다시 쏩니다.** 정체성이 유지된 재연결에서는
**PC 가 `connected`/`completed` 인 상대는 건너뛰어야** 합니다. 안 그러면 살려 둔
연결을 스스로 다시 부수게 됩니다(= 이 작업의 목적이 무너짐).

건너뛴 상대는 그대로 두고, 상태가 나쁜 상대만 기존 `vcCreatePeerAndOffer` 로 복구합니다.

### 3-C. 진짜로 미디어가 죽었다면?

**아무것도 새로 만들지 않습니다.** 이미 있는 워치독들이 개별 피어 단위로 복구합니다:

- ICE `failed`/`disconnected` → `vcReconnectPeer` (`index.html:12328` 부근)
- 검은 영상 / 프리즈 / 오디오 워치독 (`index.html:9780`, `9973` 부근)

즉 이 변경은 **"전체를 부수는 것"을 "필요한 하나만 고치는 것"으로 바꾸는 일**입니다.
복구 능력은 줄지 않습니다.

## 6. 켜고 끄는 스위치 — 이번엔 반드시 넣는다

`wrangler.toml` 에 `VC_STICKY_UID = "on" | "off"` 를 둡니다.

- **서버가 판단의 주체**입니다. `off` 면 서버는 예전처럼 새 userId 를 발급하고,
  클라이언트는 `room-joined` 의 userId 가 달라지므로 **자동으로 옛 동작**을 탑니다.
- 즉 **클라이언트 재배포 없이 서버 변수 하나로 즉시 원복**됩니다.
- ⚠️ `[vars]` 와 `[env.production.vars]` **두 곳 모두** 수정해야 합니다(이 저장소의 상습 함정).

문제가 보이면:
```bash
# wrangler.toml 두 곳을 "off" 로 바꾸고
./deploy.ps1
```

## 7. 검증 계획

### 7-1. 하네스 (`vc_sticky_reconnect_harness.mjs`, 신규)
- 서버: 옛 소켓을 닫기 **전에** `joined:false` + `stale-` 로 갱신하는가
- 서버: 인계된 재입장에서 `user-joined` 를 **쏘지 않는가**
- 서버: `VC_STICKY_UID` 가 `off` 면 예전 경로를 타는가
- 클라: `room-joined` 의 userId 가 **다르면** `vcCleanupAllPeers()` 를 부르는가(안전한 쪽 유지)
- 클라: `existing-users` 에서 `connected` 인 피어는 offer 대상에서 **제외**하는가
- 클라: 스위치가 꺼진 상태에서 기존 동작이 100% 그대로인가

### 7-2. 실측 — 하네스로는 절대 못 잡는 것
**이건 반드시 실제 2인 통화로 확인해야 합니다.**

1. PC 2대(또는 PC+폰)로 같은 방 입장, 영상·음성 정상 확인
2. 한쪽에서 **네트워크를 5초간 끊었다 붙임**(Wi-Fi 토글 / DevTools Network offline)
3. 확인할 것:
   - 상대 화면에서 내 타일이 **사라지지 않는가**
   - 영상이 **끊기지 않고 이어지는가**(끊겨도 3초 이내 복귀)
   - "재연결 중" 배너가 뜨더라도 **곧 사라지는가**
   - 채팅·교재 공유가 재연결 후 정상인가
4. 30초·60초 끊김도 동일하게 확인(20초 넘으면 유령 타일이 사라지므로 동작이 달라짐)
5. **필리핀 강사 1명과 실제 수업 1회** — 최종 판정은 여기서

### 7-3. 배포 후 근거 확인

`[VideoChat][close] ... code=1006` 빈도가 **변하지 않아야** 정상입니다

## 8. 이 작업에 **포함하지 않을 것**

한 번에 다 하려다 사고가 납니다. 아래는 별건으로 남깁니다.

- **clientId 위조 방지** — 서버가 세션 토큰으로 clientId 를 검증하는 구조.
  범위가 커서 별도 작업. (현재도 같은 위험이 있으므로 이번 작업이 만드는 신규 위험은 아님)
- **관찰자(observer) 경로 정비** — `join-observe` 가 서버 switch 에 case 가 없어
  `Unknown message type` 으로 빠짐(`video-call-room.ts:105-146`). 기존 문제.
- **끊김 이벤트 D1 적재** — `vc_quality` 와 같은 패턴으로 `vc_close_log` 신설.
  로그 샘플링(5%)의 한계를 보완하지만 이번 작업과 독립.
- **DO 서버측 유휴 소켓 정리(alarm)** — 서버에는 하트비트 타임아웃이 없음. 별건.

1. **작업 시점** — 수업이 없는 시간대(새벽)에 배포할지, 지금처럼 바로 할지

## 부록 — 지금까지 확인된 사실 (다시 조사하지 말 것)

- `createWebSocket` 호출처는 **2곳뿐**이며 둘 다 `/ws/video-call` (`index.html:11402`, `11663`)
- 서버는 `ping` 에 **join 여부와 무관하게** `pong` 응답 (`video-call-room.ts:145` → `send` → `wsOf`)
- 소켓 accept 시점에 이미 userId 부여 (`video-call-room.ts:64`)
- `remoteCount()` 는 DOM 이 아니라 `vcPeerConnections` 를 셈(`index.html:4105`)
  → 유령 타일은 수업 종료 판정에 영향 없음
- `clientId` 는 **탭 단위**(`sessionStorage`), 계정 단위가 아님 — 과거 무한 킥 루프 대응
- TURN 은 실측 검증 완료. 필리핀 튕김의 원인 아님
