# 크몽 업체 진단 검증 + 대응 정리

통화(2026-07-09) 내용을 전사해 업체 주장을 실제 코드/라이브 서버와 대조한 결과입니다.
**결론 먼저: 업체 주장 중 "관리자가 다 열려있다"는 사실이었고, 오늘 근본 수정했습니다. 그러나 "처음부터 다시 3천만원"은 과장입니다.**

---

## 1. 업체가 통화에서 주장한 것 (요지)

1. 로컬 개발환경이 깨져 있어 우리가 이어서 할 수 있는 게 없다.
2. HTML 한 파일에 4만 줄, 프레임워크 없이 통짜라 하나 건드리면 다 망가진다.
3. 백업이 없어서 비즈니스 데이터가 날아갈 위험이 크다.
4. WebSocket(“볼트”) 프로토콜을 원했는데 지금은 HTTP API라 구조가 달라 전면 재구성 필요.
5. **관리자 인증이 사실상 다 열려있어 아무나 접근 가능한 게 확인됐다.**
6. 그래서 처음부터 다시 개발해야 한다 → 기획 1~2주 + 개발 1~1.5개월 + QA = 약 2.5개월, **3천만원~**.

## 2. 각 주장에 대한 검증 판정

| # | 주장 | 판정 | 근거 |
|---|---|---|---|
| 1 | 로컬 개발환경 깨짐 | **부분사실·과장** | 로컬 `wrangler dev`가 Windows에서 크래시하는 알려진 문제는 있음. 하지만 **운영은 Cloudflare에 정상 배포되어 살아있음**(라이브 API 200 확인). "이어서 할 수 없다"는 로컬만 보고 한 말. |
| 2 | 4만 줄 통짜 HTML | **사실** | `admin.html` 40,239줄, `index.html` 25,996줄 실측. 프레임워크 없이 인라인. 유지보수 위험 높음은 맞음 → 이미 [분리 계획](REFACTOR_PLAN.md) 착수. |
| 3 | 백업 없음 | **확인 필요·과장 소지** | 코드/DB는 git + Cloudflare D1에 있음. D1 자동 백업(Time Travel) 여부는 별도 점검 필요. "다 날아간다"는 근거 부족. |
| 4 | WebSocket 재구성 필요 | **오해** | 화상수업은 **이미 WebSocket + Durable Object로 동작 중**(SignalingRoom/VideoCallRoom). 전면 재작성 근거 약함. 일반 API가 HTTP인 건 정상 설계. |
| 5 | 관리자가 다 열려있다 | **✅ 사실 — 심각** | 인증 없이 200 반환하는 `/api/admin/*` 다수 확인. 실제 유출: 학생평가·감사로그(관리자 IP 포함)·미납독촉 내역·경영브리핑·매출/이탈 예측·카톡 수신함 등. **원인·수정은 아래 3장.** |
| 6 | 처음부터 3천만원 재개발 | **과장(영업)** | 5번은 오늘 **소스 수정 한 곳**으로 근본 차단. 2번은 점진 분리로 대응 가능. 전면 재작성이 유일한 해법이라는 주장은 성립하지 않음. 재작성 시 오히려 30년 녹아든 기능 대량 회귀 위험. |

> 요약: 업체가 **진짜 문제(5번 보안, 2번 구조)를 정확히 짚은 것은 맞다.** 다만 그 해법을 "전면 재개발 3천만원"으로 몬 것은 상업적 과장이다. 진짜 문제는 재작성 없이도 고칠 수 있고, 오늘 가장 위험한 것부터 고쳤다.

## 3. 오늘 고친 것 — 관리자 API 무인증 노출 (주장 5)

### 원인
`cloudflare-deploy/src/index.ts`의 인증 게이트 `isAdminPath()`가 **허용목록(allowlist)** 방식이었음.
→ 새 `/api/admin/*` 엔드포인트를 만들 때 이 목록에 등록을 빠뜨리면, 그 API는 **인증 없이 전부 공개**됨.
→ 실제로 수십 개가 목록에서 누락돼 무인증 노출 상태였음(라이브 확인).

**노출 확인된 민감 엔드포인트(일부):**
`/api/admin/eval/list`(학생 평가·이름), `/api/admin/audit-logs`(관리자 행동+실IP),
`/api/admin/dunning/log`(미납 독촉·연체금액), `/api/admin/briefing/latest`(경영 브리핑),
`/api/admin/forecast/revenue`·`/forecast/churn`(매출·이탈 예측), `/api/admin/kpi/dashboard`,
`/api/admin/gifts/redemptions`(기프티콘 교환), `/api/admin/families`, `/api/admin/kakao/inbound`,
`/api/admin/no-shows`, `/api/admin/inquiry/list`, `/api/admin/monthly-report/list` 등.

### 수정 (default-deny 로 전환)
```
/api/admin/ 로 시작하면 → 무조건 인증 필요 (기본 차단)
   단, 학생/학부모 화면이 이미 쓰던 소수만 명시적 공개 예외(isAdminPublicApi)
```
이제 **새 admin API를 추가해도 자동으로 인증이 걸림.** 빠뜨려서 뚫리는 구조 자체를 제거.

### "기능 안 깨짐" 검증 (회귀 0 원칙)
비관리자 화면이 호출하는 admin API를 전수 조사 → **현재 이미 공개라서 막으면 깨지는 것만** 예외로 남김:

| 예외로 남긴 것 | 쓰는 화면 | 이유 | 후속 과제 |
|---|---|---|---|
| `/api/admin/points/list` | index.html 학원랭킹 | 현재 공개, 막으면 랭킹 깨짐 | 전용 `/api/points/leaderboard`(최소필드)로 대체 |
| `/api/admin/gifts/seed-catalog` | index.html 기프트샵 | 카탈로그 최초 시드 | 관리자 시드로 이관 |
| `/api/admin/ai-analyze/student` | parent.html 학부모 | 학부모 인증 체계 없음 | 학부모 토큰 인증 도입 후 잠금 |

→ 나머지 학생/강사 호출 API는 이미 401이라 변화 없음. **poster-maker(관리자 도구)의 업로드는 이제 정상적으로 인증 요구**(로그인한 관리자는 그대로 사용).

### ⚠️ 활성화하려면 배포 필요
이 수정은 **커밋만 된 상태**입니다. `deploy.ps1`을 실행해야 실제 서버에 적용됩니다.
배포 후 확인: 로그아웃 상태에서 `/api/admin/eval/list` 가 **401**이 나오면 성공(지금은 200).

## 4. 보안 후속 진행상황 (2026-07-10 업데이트)
- ✅ **공개 예외 3 → 1로 축소**: `points/list`는 전용 `/api/points/leaderboard`(최소필드)로, `gifts/seed-catalog`는 공개 카탈로그 자동 시드로 대체해 둘 다 인증 필요로 전환. 남은 예외는 `ai-analyze/student` 1개뿐.
- ✅ **`write-history` IDOR 차단**: `mango_token` 서명 uid 일치 요구(ai-write.html이 토큰 전송).
- ✅ **D1 백업 확인**: Cloudflare D1은 **30일 Time Travel(시점 복원)** 기본 내장 → "백업 없음" 주장 반박됨.
- ⏳ **`ROOM_JWT_SECRET` 설정 필요(사장님 실행)**: 지금은 공개 유추 가능한 폴백 사용 중.
  ```
  cd cloudflare-deploy
  npx wrangler@latest secret put ROOM_JWT_SECRET                  # 강한 랜덤값 입력
  npx wrangler@latest secret put ROOM_JWT_SECRET --env production # 같은 값
  ```
- ⏳ **`ai-analyze/student`**: 학부모 로그인/토큰 도입 후 잠글 것(parent.html에 인증 부재).
- 📌 **성능 참고**: D1 24시간 rows_read 4.8억 건 — 인덱스 없는 풀스캔 의심, 별도 점검 권장.

## 5. 권고
- **3천만원 전면 재개발은 보류 권장.** 진짜 문제(보안·구조)는 재작성 없이 수정 가능하고, 오늘 최우선 위험을 제거함.
- 계속할 일: (1) 배포로 보안수정 활성화 → (2) 예외 3개 정리 → (3) [분리 계획](REFACTOR_PLAN.md)대로 admin.html·api-mango 점진 분리.
- 업체와 미팅 시(월요일 예정), "관리자 무인증 노출은 이미 default-deny로 근본수정했다"는 점을 근거로 재개발 범위·금액을 다시 협상할 수 있음.
